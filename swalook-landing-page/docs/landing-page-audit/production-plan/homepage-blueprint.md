# Homepage Blueprint

## Status
- **State:** Draft production blueprint
- **Scope:** Homepage section order, CTA placement, trust/proof guidance, mobile behavior, and product-truth dependencies
- **Primary sources:** `master-roadmap.md`, `execution-pack.md`, `home-page-sections.md`, `content-ops-checklist.md`
- **Use case:** Execution-ready reference for homepage content, design, and build alignment
- **Rule:** Do not add claims, features, integrations, metrics, or testimonials that are not already supported by product truth and approved source material

## Goal
Create a homepage that:
- explains what Swalook is in the first screen
- shows the core value proposition quickly
- reduces buyer uncertainty with proof and operational credibility
- routes users to the correct next step with a clear CTA hierarchy
- stays aligned with current product truth and content operations review

## Section Order Blueprint

### 1) Hero / Primary Value Proposition
**Purpose**
- State the product in one sentence.
- Explain the main outcome the visitor can expect.
- Establish immediate relevance for the intended audience.

**Content requirements**
- Headline: outcome-led, specific, and non-promotional.
- Supporting line: clarify what Swalook does and for whom.
- Primary CTA: one dominant action only.
- Secondary CTA: optional, lower emphasis, for users who need more context.

**CTA placement**
- Primary CTA above the fold, visually dominant.
- Secondary CTA adjacent to the primary CTA, not separated into another region.

**Trust/proof guidance**
- Include only lightweight proof here: recognized brand names, product category clarity, or a short credibility line if backed by source material.
- Avoid dense testimonials or unsupported claims in the hero.

**Mobile considerations**
- Keep the headline to a short, scannable stack.
- Place CTA buttons immediately after the value statement.
- Avoid hero layouts that push CTAs below the first scroll on standard mobile heights.

**Must be backed by product truth**
- Product name, category description, and headline claim.
- Any quantified statement or “best-in-class” language.
- Any logos, customer names, or usage claim shown in the hero.

---

### 2) Trust Strip / Quick Credibility Layer
**Purpose**
- Reduce first-click hesitation after the hero.
- Signal legitimacy before deeper explanation begins.

**Content requirements**
- Only use proof types that exist in the source set: customer logos, partner marks, security indicators, or short factual validation lines.
- Keep copy minimal.

**CTA placement**
- No CTA required unless the source plan explicitly calls for one.
- If used, keep it tertiary and non-disruptive.

**Trust/proof guidance**
- This section should be factual, not persuasive.
- Do not invent client counts, ratings, certifications, or media mentions.

**Mobile considerations**
- Use a compact horizontal-to-vertical fallback.
- Ensure logos or badges do not become unreadably small.

**Must be backed by product truth**
- Every logo, badge, and trust claim.

---

### 3) Core Benefits / Value Props
**Purpose**
- Translate the product into 3–4 concrete benefits.
- Answer “why this matters” after the initial promise.

**Content requirements**
- Each card or block should contain:
  - a short title
  - one-sentence explanation
  - a proof or consequence if supported by source material
- Keep the benefits distinct and non-overlapping.

**CTA placement**
- Optional micro-CTA only if the section is long.
- Main conversion CTA should remain above and below this section, not inside every card.

**Trust/proof guidance**
- Use product outcomes only when documented.
- Avoid generic marketing language that cannot be traced to source material.

**Mobile considerations**
- Stack cards vertically.
- Keep each block short enough to scan in one screenful on mobile.

**Must be backed by product truth**
- Benefit statements.
- Any workflow, outcome, or operational improvement referenced.

---

### 4) Product Overview / What Swalook Does
**Purpose**
- Explain the product at a practical level.
- Make the operating model understandable for visitors who need more detail.

**Content requirements**
- Show how the product works at a high level.
- Use plain language over feature jargon.
- If the homepage includes modules or capabilities, name only those already defined in the source docs.

**CTA placement**
- No required CTA.
- If included, use a contextual “See how it works” style action linked to a deeper page.

**Trust/proof guidance**
- Pair explanation with a factual visual or process cue if available.
- Do not imply capabilities that are not described in the product documentation.

**Mobile considerations**
- Prefer a simple vertical flow.
- Reduce the number of columns and avoid text-heavy sidebars.

**Must be backed by product truth**
- Feature names, workflow descriptions, and product architecture references.

---

### 5) Featured Capabilities / Use Cases
**Purpose**
- Show the most important things the visitor can do with the product.
- Help users self-identify whether the product fits their needs.

**Content requirements**
- Select the few capabilities that matter most to the homepage audience.
- Keep each capability tied to a user problem or job-to-be-done.
- If use cases are shown, they must reflect current product support.

**CTA placement**
- Contextual CTA may be used after the capability grid, especially if there are multiple content blocks.
- Avoid CTA spam inside each card.

**Trust/proof guidance**
- If a capability is complex, add a factual cue that demonstrates maturity rather than hype.
- Do not use aspirational “coming soon” language on the homepage.

**Mobile considerations**
- One card per row on mobile.
- Keep visual assets lightweight and consistent.

**Must be backed by product truth**
- Every capability and use case.
- Any screenshot, workflow, or module label.

---

### 6) Proof / Evidence / Outcomes
**Purpose**
- Convert interest into confidence.
- Demonstrate that the product delivers measurable or observable value.

**Content requirements**
- Use evidence in the strongest available order:
  1. verified customer quote
  2. documented result
  3. case study reference
  4. product metric
  5. internal operational proof only if external proof is unavailable
- Keep proof concise and attributable.

**CTA placement**
- Optional CTA after proof if this is the deepest persuasion block.
- If the proof section is long, place the next conversion CTA after it.

**Trust/proof guidance**
- Every quote must be attributed and approved.
- Every metric must be sourced and current.
- Do not combine multiple unsupported claims into a single “proof” block.

**Mobile considerations**
- Use one proof item per block.
- Ensure quote attribution stays visible and readable.

**Must be backed by product truth**
- Customer names, job titles, metrics, dates, and case references.
- Any testimonial or outcome language.

---

### 7) How It Works / Process Overview
**Purpose**
- Reduce friction by showing the user journey.
- Clarify what happens after clicking the CTA.

**Content requirements**
- Use 3 steps max unless the source docs explicitly require more.
- Keep steps action-oriented and aligned to actual onboarding or usage flow.
- Include any handoff or implementation step only if it exists in the current process.

**CTA placement**
- Good place for a “Get started” or “Talk to sales” CTA if the process is a major decision point.
- Make the CTA reflect the actual next step in the funnel.

**Trust/proof guidance**
- The process must match actual product and operational flow.
- Avoid pretending self-serve is available if it is not.

**Mobile considerations**
- Vertical step layout.
- Numbered steps should remain readable without horizontal scrolling.

**Must be backed by product truth**
- Step names, onboarding flow, delivery sequence, and any required inputs.

---

### 8) FAQ / Objection Handling
**Purpose**
- Answer the questions that block conversion.
- Capture concerns that repeat across sales, support, or audit feedback.

**Content requirements**
- Prioritize genuine objections, not filler questions.
- Keep answers short, specific, and operationally accurate.
- Include only questions that are validated by product truth or recurring user demand.

**CTA placement**
- Optional final CTA after FAQ.
- CTA should invite the next step, not restate the entire value prop.

**Trust/proof guidance**
- FAQs must not contradict the source docs.
- If something is not finalized, say so clearly rather than over-promising.

**Mobile considerations**
- Accordion behavior is recommended for readability.
- Make question labels short enough for one-line display where possible.

**Must be backed by product truth**
- All answers, especially pricing, implementation, support, and capability questions.

---

### 9) Final CTA / Conversion Close
**Purpose**
- Give users a clear final decision point.
- Capture intent from visitors who have scanned the full page.

**Content requirements**
- Repeat the strongest value proposition in shorter form.
- Use one primary CTA only.
- Keep supporting copy minimal and reassuring.

**CTA placement**
- End of page, visually distinct from preceding content.
- If a form is present, keep it short and aligned with the actual funnel.

**Trust/proof guidance**
- Support the CTA with one final factual reassurance, not a new claim.
- Do not introduce new proof types here that were not already used above.

**Mobile considerations**
- Keep CTA visible without extra scrolling at the end of the page.
- Make the final block large enough for thumb interaction.

**Must be backed by product truth**
- CTA destination, form fields, and any promise about response time or next steps.

## Global CTA Hierarchy
1. **Primary CTA:** one action that matches the main conversion goal.
2. **Secondary CTA:** one lower-friction option for visitors who are not ready to convert.
3. **Contextual CTAs:** only where they support comprehension or a documented funnel step.

Rules:
- Do not allow multiple competing primary CTAs on the same page.
- Keep CTA labels action-oriented and specific.
- Make CTA destinations consistent with the actual funnel path defined in the source docs.

## Trust and Proof Rules
- Prefer proof that can be independently verified.
- Do not mix aspirational messaging with factual proof in the same line.
- Any customer logo, testimonial, quote, metric, certification, or partner reference must be cleared against source material.
- If proof is incomplete, use restrained language and defer the stronger claim until the source is available.

## Mobile Implementation Notes
- Design for a single-column reading flow on small screens.
- Keep paragraphs short and avoid long supporting copy.
- Preserve CTA visibility within the first viewport and again near the end.
- Ensure logos, screenshots, and proof elements remain legible when stacked.
- Avoid layouts that rely on hover, tight side-by-side comparison, or dense icon rows.

## Product Truth Requirements
The homepage must be grounded in the following categories of truth:
- what Swalook is
- who it is for
- what problems it solves
- what the product actually does today
- what proof is currently approved
- what the real next step in the funnel is

Any content outside those categories needs explicit confirmation before implementation.

## Implementation Risks
- **Unsupported claim risk:** homepage copy may drift into marketing language not present in the source docs.
- **Proof mismatch risk:** testimonials, logos, or metrics may be used without validation.
- **CTA ambiguity risk:** multiple conversion targets can dilute the intended action.
- **Mobile density risk:** too many sections or wide layouts can reduce scannability on smaller screens.
- **Process accuracy risk:** onboarding or workflow descriptions may not match the actual product flow.
- **Content drift risk:** section order may be changed during build in ways that break the approved narrative.

## Build Guidance
- Treat this blueprint as the homepage content contract.
- Use it to align design, copy, and implementation before final production work.
- If a section cannot be supported by product truth, remove it rather than inventing content.
